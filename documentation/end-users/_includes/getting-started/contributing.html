Contributing

We welcome contributions to the BAS Style Kit from anyone at any skill level.

A contribution may include things like documentation changes or improvements through to a fundamental refactoring of grid styles or adding a new custom component.

Issue tracking

For reporting bugs and issues, proposing changes or new additions or for any other general feedback please use our issue tracker, Jira.

To quickly report an issue to through Jira, use the issue submission form.

If reporting an issue through Jira directly please use the BAS Style Kit project.

Discussions

For more general feedback or for discussions on best practice etc. where there is no clear or "right" answer, please add a topic to this document.

We are actively looking into better ways of managing these sorts of discussions so this will improve in the future.

Audience

Contributions will be accepted from anyone, though it is a assumed you are a BAS employee.
Other people who may make contributions are contractors for websites, such as those used for the new public website.

Contributions from BAS staff or non-BAS staff are treated in exactly the same way.

Protected aspects

These are elements of the Style Kit which we likely won't change, for either technical, design or branding reasons. They include:

  • Anything relating the BAS logo - though this can be discussed, the usage and design of the BAS logo is controlled by NERC
  • Anything relating to the NERC logo and its branding - similar to the above, this is controlled by NERC
  • Font choices and colour schemes - these are design and/or branding decisions and are therefore subject to personal taste
  • Fundamental aspects of Bootstrap - this includes things like the grid system, vertical rhythm and component layout

Development environments

To make changes to the underlying code that makes up the BAS Style kit, such as its Less styles, you will need certain tools available on your computer. The main project README (/README.md) for this proejct contains detailed instructions on how to setup such an environment, locally or remotely.

It also contains instructions and guidance on the various Gulp tasks used for processing Less to CSS and copying fonts etc.

Pull requests

If you have made changes within a development environment you wish to contribute back to this project, please use the following process, which uses the Git Flow branching model.

Ideally pull requests should be made in response to an issue from Jira, so that the wider context is known. However this project will also accept unexpected pull requests where this makes sense.

Changes will be merged into the next release under the project's normal timeline.
If your change is urgent ensure this is clear in your pull request, or make a hotfix branch, instead of a feature branch described below.

  1. Before you modify the Style Kit, create a feature branch in the project repository, where possible do this from a Jira issue for automatic linking
  2. Make your changes, making sure to update documentation where needed
  3. Update the Changelog using the keep-a-changelog methodology
  4. Create a pull request to the develop branch of the repository,
    • Notify Felix Fennell that you have done this
    • The pull-request will be reviewed manually to ensure the changes made won't cause an side-effects and required documentation changes have been made
    • Comments may be left for to correcting things like naming conventions or other minor issues
    • When these are resolved the changes will be merged into develop in readiness for the next release
    • If there is a fundamental issue with your pull request may be left open for a wider discussion and then accepted or rejected accordingly
    • In extreme cases your pull request may be rejected, in which case suitable reasoning will be supplied to explain why this happend
